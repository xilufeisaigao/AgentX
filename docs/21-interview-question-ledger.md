## 2026-03-26

### 题目 1

- 来源: 系统重构后的默认题
- 主题: 项目总述
- 状态: 已确认
- 问题: 这个项目最大的技术亮点是什么，为什么不是简单的 prompt 工程？
- 当前推荐答案:
  AgentX 的亮点不在于“把模型接进来”，而在于把 Agent 约束成受控的工程执行单元。项目里真正落地的是状态机、上下文快照、Worker 调度、Merge Gate 和 HITL 这些控制面机制，所以它解决的是长流程交付里的幻觉、遗忘、越权和不可追责问题，而不是单次生成质量问题。

### 题目 2

- 来源: 系统重构后的默认题
- 主题: 记忆设计
- 状态: 已确认
- 问题: 你们的记忆模块是怎么设计的，为什么不用聊天历史直接续写？
- 当前推荐答案:
  我们没有把记忆当成聊天历史缓存，而是拆成确认需求、ticket 事件、run 证据和 repo context 四层事实，再由 `ContextCompileService` 编译成 `task_context_snapshots`。这样每次 run 绑定的是最新 `READY` 快照，而不是一长串可能过时的聊天记录，更适合长流程工程场景。

### 题目 3

- 来源: 系统重构后的默认题
- 主题: RAG
- 状态: 已确认
- 问题: RAG 在这个项目里承担什么角色，为什么还需要上下文快照？
- 当前推荐答案:
  RAG 负责代码上下文召回，当前实现是 semantic retrieval 优先、lexical fallback 兜底；但它只能解决“找得到什么”，不能解决“当前有效的事实是什么”。上下文快照负责把确认需求、ticket 事件和运行证据裁决成唯一有效输入，所以两者是召回和裁决的分工关系。

### 题目 4

- 来源: 正式学习轮次 I0-R1
- 主题: 项目总述
- 状态: 待修正
- 问题: 如果面试官让你先用 30 秒介绍 AgentX，你会怎么说？
- 当前推荐答案:
  AgentX 是一个面向长流程软件交付的 Agent 控制面系统。它不是让模型自由写代码，而是通过模块化状态机、上下文快照、Worker 调度、Merge Gate 和工单化 HITL，把需求、决策、执行、验证和交付串成一个可审计的闭环。它主要解决的是长会话工程场景里的幻觉、遗忘、越权和不可追责问题。
