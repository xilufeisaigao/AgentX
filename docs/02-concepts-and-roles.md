# 模块 02：项目概念与角色定义

范围说明：
本文件专门用于沉淀 AgentX 的“概念（术语）”与“角色（Agents/Roles）”定义，便于后续文档引用与追溯设计思路。

非目标：
本文件不编排完整工作流状态机，不定义需求文档契约的最终格式，不展开各阶段的执行细节（这些在后续模块逐步确定）。

## 1. 概念（术语）定义（暂定）

说明：
以下定义以“可控、可审计、可回滚、可交付”为前提，所有术语都应能落到明确输入/输出/责任边界。

1. AgentX（系统）
   - 定义：一个将“软件交付流程”自动化的系统，而不是“自动写代码”的系统。
2. 交付（Delivery）
   - 定义：一组可运行、可验证、可回滚的变更结果，其载体必须能被唯一定位与复现（例如一个 git commit 或等价的不可变标识）。
3. 需求文档（Requirement Document）
   - 定义：由需求 Agent 输出、由用户确认的“需求基线文本”。
   - 备注：格式/字段/契约后续单独定义；在格式未定前，仍要求其具备“可审查、可确认、可版本化”的最小性质。
4. 契约（Contract）
   - 定义：把自然语言意图变成“可检查的输入/输出约束”的结构化约定。
   - 目标：降低提示词依赖，提升可验证性与可控性。
5. 决策点（Decision Point）
   - 定义：必须由人类作出高价值取舍的节点（例如范围取舍、风险接受、是否进入下一阶段、是否停止自动推进）。
6. 决策面（Decision Surface）
   - 定义：系统对人类开放的一组“可枚举、可审计、可回放”的决策点集合。
   - 目标：保证“人不写代码”仍然能“控方向、控风险、控停止”。
7. 逻辑架构（Logical Architecture）
   - 定义：与技术栈无关的系统结构描述，包含模块边界、职责分配、数据流转与交互逻辑（谁与谁交互、以何种契约交互、失败如何传播）。
8. 技术栈画像（Stack Profile）
   - 定义：对“实现约束”的参数化描述，例如：语言/框架、数据库类型、消息系统、部署形态、运行环境约束等。
   - 规则：Stack Profile 只能影响“实现策略”，不应改变“逻辑边界与数据语义”。
9. 数据模型（Data Model）
   - 定义：对业务/领域概念的数据表达，包括实体、关系、约束、不变量、生命周期状态与访问模式（读写路径）。
10. 表结构（Schema）
   - 定义：数据模型在关系型数据库中的落地表达，包含表/字段、主键、唯一性约束、索引、外键（如适用）、以及演进方式（迁移策略）。
11. 可验证性函数（Fitness Function）
   - 定义：把关键架构约束转成“可自动检查的规则”的机制（例如依赖方向约束、跨域调用禁令、必须幂等、必须审计事件等）。
   - 目标：从“感觉正确”变成“可证明不越界”。
12. ADR（Architecture Decision Record）
   - 定义：架构关键决策记录，至少包含：问题、备选方案、取舍理由、结论、未决项与复盘入口。
13. 价值层约束（Value Constraints）
   - 定义：用户在“价值/目标/范围/优先级/风险接受/验收标准”层面提出的约束。
   - 直白理解：回答“我们要做什么、为什么做、做到什么程度算成功”，而不是“具体怎么实现”。
14. 架构层问题/变化（Architecture-Level Change）
   - 定义：会改变系统边界、数据语义、核心数据流转、关键非功能属性实现方式的变化或问题。
   - 直白理解：回答“系统怎么分块、数据怎么流、关键能力怎么保证”，但仍不涉及具体代码。
15. 提请（Decision Request / Proposal）
   - 定义：把一个需要人类选择的架构/实现方式问题，整理成一份“可选择、可比较、可记录”的问题单。
   - 最小结构：问题是什么、可选方案有哪些、每个方案的优缺点/成本/风险、推荐默认、需要用户确认的点。
16. 移交（Handoff）
   - 定义：把一个问题从一个 Agent 的职责范围内，连同必要上下文与约束，一次性转交给另一个 Agent 处理。
   - 目标：减少“重复解释”，避免丢失关键约束，保证后续可追溯。
17. 会话连接（Session Link / Continuous Context）
   - 定义：多个 Agent 在同一个“项目会话”里工作，能共享同一份需求文档版本、决策记录与产物索引，并能随时把问题移交给对方。
   - 直白理解：不是让 Agent 之间真的“开着语音一直聊”，而是共享同一本“项目账本”，并能互相派单。

## 2. 角色（Roles/Agents）定义（暂定）

说明：
角色的“职责”与“边界”必须清晰，以避免上下文同质化与任务越界导致的不可控。

1. 用户（User）
   - 定义：唯一的需求与价值主语。
   - 权限：做高价值决策（提交/确认需求、接受风险、范围取舍、是否继续推进等）。
   - 约束：不参与任何具体代码编写。
2. 需求 Agent（Requirement Agent）
   - 定义：与用户交互，形成并迭代“需求文档”的 Agent。
   - 目标：把自然语言需求收敛为“价值层约束明确、可确认、可版本化”的需求文档，并持续维护其一致性。
   - 边界：只负责价值层面的约束维护；当用户的修改/问题涉及架构层问题/变化时，必须移交给架构师 Agent（以提请的形式推进）。
3. 架构师 Agent（Architect Agent）
   - 定义：在不编写代码的前提下，把需求文档中的价值约束翻译成“逻辑架构 + 数据模型/表结构建议”，并将不确定或需要取舍的点整理成提请，交由用户做高价值决策。

备注：
后续可能还会引入规划/执行/质量/合并等角色，但在当前阶段不在本文件展开。

## 3. 架构师 Agent（Architect Agent）角色定义（建议版本 v0）

### 3.1 角色目标

架构师 Agent 的目标不是“写一份看起来合理的架构说明”，而是输出一组可审查、可追溯、可验证、可被后续角色直接消费的架构产物：
1. 固定系统的逻辑边界与数据语义，避免后续执行阶段的随机漂移。
2. 让关键取舍变成显式决策点，确保人类能只做高价值决策。
3. 让“合理性/可靠性”主要来自可验证约束，而非提示词说服力。

### 3.2 输入（Inputs）

1. 已确认的需求文档（格式契约 TBD，但必须是用户确认过的版本）
2. Stack Profile（技术栈画像，作为约束参数输入）
3. 非功能需求（如可用性、性能、审计、合规、保留期等，若缺失则必须发起澄清）
4. 现有基线与约束（若是对既有项目增量：当前交付基线、已有模块、已有数据库等）

### 3.3 输出（Outputs）

架构师 Agent 的输出必须可被“下一环节”直接使用，而不是仅供阅读的散文式描述：

1. 逻辑架构规格（Stack-Agnostic）
   - 模块/边界：每个模块的职责、输入输出、依赖方向
   - 数据流转：关键数据在模块间如何流动
   - 交互逻辑：关键用户路径对应的系统交互（同步/异步、错误传播策略）
2. 数据模型规格（优先于表结构细节）
   - 核心实体与关系（含基数、可选性）
   - 不变量与约束（唯一性、幂等语义、状态机/生命周期）
   - 访问模式（主要读写路径，用于指导索引与冗余策略）
3. 表结构建议（在关系型数据库场景）
   - 表/字段草案 + 主键/唯一约束/必要索引（强调“为什么这样设计”）
   - 演进策略草案（迁移/向后兼容/回填思路）
4. 决策清单（Decision List）
   - 将必须由人类拍板的点列清楚：每个决策的备选方案、取舍、推荐默认与风险
5. ADR 集合（Architecture Decision Records）
   - 对关键取舍形成可追溯记录，作为后续复盘入口
6. 架构提请单（Decision Requests）
   - 当出现“需要取舍/不确定”的点时，按提请结构输出，方便用户在同一会话内直接做决定

### 3.4 责任边界（Responsibilities / Non-Responsibilities）

架构师 Agent 负责：
1. 解释并固化边界：确保逻辑边界与数据语义明确且一致
2. 显式化决策：把“不可避免的取舍”变成决策面的一部分
3. 约束化产物：尽可能把关键架构约束转为可验证规则（Fitness Functions 的候选项）
4. 可追溯性：提供从“需求 -> 交互 -> 模块/API -> 数据模型/表结构”的映射线索
5. 接收移交：当需求 Agent 判定为架构层问题/变化时，接收移交并以“提请”方式推动用户决策

架构师 Agent 不负责：
1. 编写任何具体代码（含实现代码与脚手架代码）
2. 在缺失关键输入时“脑补需求”并继续推进（必须澄清/阻断）
3. 随意更改 Stack Profile 或技术选型（只能在决策清单中提出建议，由用户拍板）

### 3.5 架构师怎么干活（白话版）

目标：
把“用户想要什么（价值约束）”变成“系统应该怎么设计（架构与数据）”，并且把需要人拍板的地方变成清晰的选择题。

可以把架构师 Agent 想象成做三件事：
1. 把需求拆成“系统模块怎么分、数据怎么流、哪些点最容易出事故”
2. 把数据想清楚：到底要存什么、哪些东西必须唯一、哪些状态必须可追溯
3. 把争议点变成“提请”：给出几种实现方式，每种的好处/坏处/成本/风险，让用户选

典型工作步骤（不涉及代码）：
1. 读需求文档：先只找“硬约束”和“验收标准”，不急着画图。
2. 标记不清楚的地方：哪些信息缺了就没法设计，直接问用户补齐（这也是一种提请，但属于澄清提请）。
3. 画最小结构：把系统先粗分成几个块，每块负责什么输入输出（这是逻辑架构）。
4. 走一遍用户交互：按用户路径把“每一步要读/写哪些数据、要调用哪些模块”写出来（这是交互与数据流转）。
5. 推导数据模型：根据“必须存在的数据”和“不变量”推导实体与关系，再推导出表结构草案与必要约束/索引。
6. 产出提请：凡是存在多种实现方式且会影响成本/风险/质量的点，都整理成提请（方案 A/B/C + 优劣对比 + 推荐默认）。
7. 等用户决策后收口：把用户选择写入 ADR，并把架构规格更新到一个可复用的版本。

### 3.6 设计理由（为什么这样定义）

本角色定义背后的设计理由，是把“可靠性”从提示词转移到结构化约束与可检查产物上，避免系统再次进入不可控状态。

1. 为什么要把“逻辑架构”和“Stack Profile”分离
   - 问题：技术栈经常变化，若架构输出与栈强绑定，会导致需求与架构在迁移时重写，且容易出现“产出偏离真实需求”的漂移。
   - 设计：逻辑架构描述稳定边界与数据语义；Stack Profile 只作为实现约束输入，改变的是实现策略而不是边界本体。
2. 为什么把“决策清单/ADR”作为强制输出
   - 问题：你要求“人不写代码但要控方向”，这就需要一个明确的决策面；否则人类只能通过反复聊天隐式表达偏好，系统不可审计、不可回放。
   - 设计：把关键取舍显式化，形成可枚举的决策点；同时用 ADR 固化取舍理由，便于追溯与复盘。
3. 为什么“数据模型规格”优先于“直接生成表结构”
   - 问题：表结构如果脱离不变量与访问模式，很容易出现“字段堆砌”或无法演进的设计，后续迭代会极不稳定。
   - 设计：先定义实体/关系/约束/生命周期/访问模式，再推导表结构与索引，这是把合理性落到可解释的因果链上。
4. 为什么要强调“可追溯性映射”
   - 问题：架构文档经常“讲得对但落不下来”，本质是缺少从需求到实现的可追溯链路，导致后续角色无法验证“是否覆盖了需求”。
   - 设计：要求输出最小映射线索，后续可用于 QA/验收契约以及审计复盘。
5. 为什么引入 Fitness Function（候选项）概念
   - 问题：仅靠文档与提示词很难持续约束执行阶段的越界行为，尤其在并发与重试场景会放大偏差。
   - 设计：把架构约束转成可自动检查的规则候选项，为后续“门禁/阻断/可控性”打基础。

## 4. 需求 Agent 与 架构师 Agent 如何交互（移交与提请机制）

目标：
把你的三条确认落成“可执行的协作规则”，避免两个 Agent 职责打架或互相覆盖。

### 4.1 职责分界（最简单可落地的判定）

1. 需求 Agent 处理的内容（价值层约束）
   - 用户想达成的目标是什么
   - 范围/优先级/取舍（做什么、不做什么）
   - 验收标准（做到什么算完成）
   - 风险与成本的接受程度（例如是否允许性能换一致性）
2. 架构师 Agent 处理的内容（架构层问题/变化）
   - 系统拆成哪些模块，模块之间怎么交互
   - 数据怎么流转、怎么存、如何保证一致性/审计/可回滚
   - “实现方式”的选择题（例如同步/异步、单库/分库、事件驱动/请求驱动），但不写具体代码

说明：
很多“价值约束”会影响架构（例如“必须可审计”），此时做法是：
需求 Agent 负责把它写进需求文档；同时移交给架构师 Agent，让架构师把它转化为架构约束与提请。

### 4.2 标准移交流程（需求 -> 架构）

触发条件：
当用户的新增/修改/问题被判定为“架构层问题/变化”，或者价值约束对架构影响重大时触发。

流程（建议固定成平台规则）：
1. 需求 Agent 创建移交包（Handoff Packet）
   - 必含信息：需求文档版本、用户原话/变更点、相关价值约束、希望解决的问题
   - 可选信息：时间/成本偏好、必须避免的风险、已做过的决策（若有）
2. 架构师 Agent 接收移交包，输出“提请”
   - 提请里必须给出：2-3 个可选方案 + 每个方案的优缺点 + 推荐默认 + 需要用户确认的问题
3. 用户做决策
   - 用户可以选一个方案，也可以强制指定（系统记录为“用户指定”，并记录风险提示已告知）
4. 决策回写与对齐
   - 架构师 Agent：把决策写入 ADR，并更新逻辑架构/数据模型规格
   - 需求 Agent：把决策结果以“引用/约束”的方式体现在需求文档中，保证价值层约束与架构选择一致

关键点：
移交不是“把问题甩出去”，而是用一个结构化包把上下文一次性讲清楚；提请不是“给建议”，而是让用户能在几分钟内做出明确选择。

### 4.3 用户主动提出架构问题时（用户 -> 架构）

规则：
用户提出的架构层面问题，等价于一次移交触发。

流程：
1. 需求 Agent 识别为架构问题，生成移交包（或直接转述用户原话作为移交包的一部分）
2. 架构师 Agent 以提请形式回应（方案对比）
3. 用户决策后，按 4.2 的“回写与对齐”收口

### 4.4 需求文档确认后的修改规则（你确认的要求落地）

1. 用户确认第一版需求文档后，后续修改默认仍由需求 Agent 接收与维护
2. 若修改仅涉及价值层约束：需求 Agent 直接更新需求文档版本
3. 若修改涉及架构层问题/变化：需求 Agent 不直接“自己决定怎么做”，而是移交给架构师 Agent 出提请，让用户决策

## 5. Agent 之间如何保持“一直连接”（随时移交）的工业界参考

你提到的“保持一直连接”，在工程上通常不是指 Agent 之间保持某个长连接，而是三件事：
1. 共享同一份项目上下文（需求版本、决策记录、架构产物）
2. 任意时刻可以发起移交（派单）
3. 处理结果能回写并可追溯（像公司里工单闭环）

### 5.1 工业界常见的成熟模式（从简单到强）

1. 工单/派单模式（最像公司运作，但更硬约束）
   - 思路：把“移交”当作一张工单，工单有编号、有负责人、有输入输出、有状态（待处理/已提请/已决策/已完成）。
   - 优点：最直观、最可追溯、方便审计与复盘。
2. 工作流引擎（适合长流程、可暂停、可重放）
   - 思路：用成熟的工作流系统来管理“谁在什么时候做哪一步”，并保证崩了也能从状态继续。
   - 代表：Temporal/Cadence 类（“持久化工作流”）、Camunda/Zeebe 类（BPMN 工作流）。
   - 价值：天然支持“等待用户决策”这种卡点，并能保证不会丢状态。
3. 事件总线/消息队列（适合多模块解耦与异步协作）
   - 思路：每次移交/提请/决策都发成事件，订阅方（对应 Agent）接收后处理，再发回结果事件。
   - 代表：Kafka、RabbitMQ 等。
   - 价值：扩展性强，适合后续多 Agent 并行，但需要更严格的事件规范与幂等处理。

### 5.2 AgentX 里“持续连接”的最小落地建议（不依赖具体技术栈）

如果目标是尽快把规则跑通，最小可落地形态可以是：
1. 一个“项目账本”（单一可信存储）
   - 存：需求文档版本、提请、决策、ADR、架构规格索引
2. 一个“派单通道”
   - 任何 Agent 都能创建移交包并指派给另一个 Agent
3. 统一编号（可追溯的三类 ID）
   - `session_id`：同一个项目会话
   - `handoff_id`：一次移交
   - `decision_id`：一次提请与决策

说明：
这三件事具备了，“需求随时移交架构师”“后续 Agent 之间互相移交”就都能用同一种机制实现。
