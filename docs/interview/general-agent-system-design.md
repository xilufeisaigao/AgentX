# General Agent System Design Interview Bank

本文归档更通用的 Agent 系统设计、产品设计和工程治理高频题。这类问题不要求绑定 AgentX 当前代码真相，但答案仍然要尽量工程化、结构化，适合高频面试复用。

## Agent 定义与边界

### [重要程度：高] 你怎么理解 Agent？它和 Chatbot、Workflow、Copilot 有什么区别？

回答这题时，最重要的是不要把 Agent 说成“更聪明的聊天机器人”。

更准确的理解是：Agent 是一个围绕目标持续做决策的系统。它不只是根据输入生成回复，而是会结合当前状态、可用工具和外部反馈，决定下一步该做什么动作。

因此它至少有四个组成部分：

1. 目标
   - 它要完成什么任务
2. 状态
   - 它当前知道什么、进展到哪里、失败过什么
3. 动作
   - 它能调用哪些工具、执行哪些操作
4. 策略
   - 它如何根据状态和反馈选择动作

和几个相近概念相比：

`Chatbot`

1. 更偏对话响应
2. 通常核心问题是“这一轮该怎么回答”
3. 不一定有显式状态和动作系统

`Workflow`

1. 更偏预定义流程执行
2. 下一步通常由流程图写死
3. 灵活性较低，但稳定性高

`Copilot`

1. 仍然是人主导、AI 辅助
2. AI 提供建议，但关键决策权在人
3. 更像高质量辅助器，而不是目标驱动执行器

`Agent`

1. 更强调目标驱动
2. 更强调显式状态
3. 更强调工具使用
4. 更强调根据反馈调整策略

一句适合面试的总结是：

“Chatbot 在回答问题，Workflow 在执行流程，Copilot 在辅助人，而 Agent 在围绕目标做持续决策。”

### [重要程度：高] 什么场景适合引入 Agent，什么场景不适合？

这个问题考的是边界感。

适合引入 Agent 的场景通常有几个特点：

1. 目标明确但路径不完全固定
2. 需要多步决策，而不是单步输出
3. 需要结合状态和反馈持续调整
4. 需要调用多个工具或外部系统
5. 任务天然会出现分支、例外和动态信息

典型适合的场景包括：

1. 客服问题分诊与处理
2. 研发任务辅助推进
3. 销售线索筛选与跟进建议
4. 学习辅导与个性化推荐
5. 多系统之间的任务编排

不适合的场景也很明确：

1. 规则完全固定
2. 输入输出关系很稳定
3. 不需要动态决策
4. 高风险且无法容忍不确定性
5. 一个简单脚本或传统工作流就能稳定完成

例如：

1. 标准化报表导出
2. 固定审批流
3. 强一致账务操作
4. 高风险医疗/法律结论的最终裁定

所以最稳的说法是：

“Agent 适合的是目标明确但路径不完全固定、需要多步决策和工具协作的场景；如果事情本身是稳定规则问题，传统 workflow 往往更合适。”

## Agent 系统设计

### [重要程度：高] 如果让你设计一个 Agent 系统，你会怎么拆？

这题建议直接按系统分层来答，不要只讲 prompt。

一个比较稳的拆法是六层：

1. 目标层
   - 系统最终要完成什么
2. 状态层
   - 当前上下文、历史、环境、任务状态
3. 决策层
   - 当前应该采取什么动作
4. 工具层
   - 可调用的外部能力
5. 安全层
   - 权限、约束、回退、审计
6. 评估层
   - 怎么判断做得好不好

展开来说：

第一，目标层。  
先定义 Agent 的目标，不然系统会很容易从“完成任务”漂到“尽量多回答”。目标最好可观测，例如：

1. 完成某类工单
2. 帮用户走完一个闭环
3. 提高某个流程转化率
4. 降低人工介入率

第二，状态层。  
很多 Agent 做不好，不是因为模型不够强，而是因为状态设计太弱。至少要区分：

1. 用户状态
2. 任务状态
3. 环境状态
4. 历史动作
5. 当前上下文窗口内的工作记忆

第三，决策层。  
这里的核心不是“模型说什么”，而是“下一步做什么”。常见动作包括：

1. 提问澄清
2. 调用工具
3. 检索信息
4. 生成计划
5. 输出结果
6. 升级人工

第四，工具层。  
工具应该被明确建模，而不是让模型自由发挥。每个工具至少要有：

1. 输入约束
2. 输出结构
3. 权限边界
4. 失败处理

第五，安全层。  
这里要考虑：

1. 什么动作允许自动执行
2. 什么动作必须人工确认
3. 什么场景必须降级
4. 日志和审计如何保留

第六，评估层。  
系统上线前后要能衡量：

1. 成功率
2. 错误率
3. 人工接管率
4. 用户满意度
5. 核心业务指标变化

一句话总结：

“设计 Agent 系统时，应该把重点放在目标、状态、动作和边界，而不是只放在 prompt 上。”

### [重要程度：高] 从学术和工业角度看，一个 Agent 到底由哪些部分组成？又该怎么系统性减少幻觉？如果结合 AgentX，你会怎么回答？

这题最好分三层答，不要一上来只讲 prompt engineering。

第一层先讲学术定义。  
如果按经典 AI 视角，Agent 本质上是一个“在环境中感知、决策并行动，以追求某个目标”的系统。比较完整地拆，至少有六个部分：

1. `环境`
   - Agent 所处的外部世界，例如代码仓库、数据库、用户、第三方系统、运行环境
2. `感知`
   - Agent 怎么拿到观察，例如对话输入、检索结果、文件内容、日志、工具返回
3. `内部状态`
   - Agent 当前记住什么，包括任务状态、历史动作、上下文窗口、工作记忆、显式持久化状态
4. `决策/策略`
   - Agent 如何根据目标和状态选择下一步动作，可能表现为 policy、planner、reasoner 或 controller
5. `动作/执行器`
   - Agent 能做什么，例如调用工具、写文件、发请求、提问澄清、升级人工
6. `目标/效用函数`
   - 系统最终追求什么，怎么判断这一步是不是在朝正确方向前进

如果面试官继续深挖，还可以补第七层：

1. `学习/评估回路`
   - 用于在长期运行里修正策略、沉淀经验、做回归评测

所以从学术上讲，Agent 不是“一个会聊天的大模型”，而是“目标 + 状态 + 感知 + 决策 + 动作 + 反馈”的闭环系统。

第二层再讲“幻觉为什么出现”。  
幻觉本质上不是单点问题，而是 Agent 在这些环节里任意一层失真：

1. 感知拿到的是错的、旧的、不完整的事实
2. 内部状态混乱，把历史对话、任务状态和业务真相混在一起
3. 决策层没有边界，证据不足时还被迫继续生成
4. 动作层过于自由，没有工具约束和权限约束
5. 目标函数不清，只追求“回答得像样”，而不是“事实正确、可验证、可执行”

因此减少幻觉，不能只靠一句“请谨慎回答”，而要系统化治理。

第三层再讲工业情境下怎么做。  
工业里更稳的做法通常是六件事一起上：

1. `结构化真相优先`
   - 核心状态、规则、配置、任务边界优先走结构化存储和精确查询，不让模型从自由文本里猜
2. `把感知和动作工具化`
   - 让模型通过受控工具读真实世界，而不是靠参数记忆脑补
3. `显式状态机`
   - 把 requirement、ticket、task、run、workspace 这类状态单独建模，避免全塞进聊天历史
4. `输出和动作都收口`
   - 用结构化输出、typed tool call、参数校验、权限边界，减少自由发挥空间
5. `证据不足就降级`
   - 可以 ASK_BLOCKER、追问澄清、转人工，也不要在证据不足时硬编
6. `验证和评测闭环`
   - 不只看模型回答，还要看工具证据、运行结果、测试结果、回放和回归评测

如果把它压缩成一句适合面试的话，就是：

“工业场景里减少幻觉，核心不是让模型更保守，而是把真相来源、状态、工具、权限、验证和人工介入边界都显式化。”

最后再落到 AgentX。  
结合我们的项目，这个问题可以直接映射到固定主链和三层架构上。

第一，`目标层` 在 AgentX 里不是抽象聊天目标，而是固定主链：

`requirement -> architect -> ticket-gate -> task-graph -> worker-manager -> coding -> merge-gate -> integration-test-gate -> verify`

这意味着平台不是追求“多说点合理的话”，而是追求“沿固定主链把交付推进到真正可验证完成”。

第二，`内部状态层` 在 AgentX 里不是 chat memory，而是结构化真相。  
当前最关键的状态对象包括：

1. `RequirementDoc`
2. `Ticket`
3. `WorkTask`
4. `TaskContextSnapshot`
5. `TaskRun`
6. `GitWorkspace`
7. `WorkflowRun / WorkflowNodeRun`

这也是为什么我们一直强调：`TaskRun.SUCCEEDED != WorkTask.DONE`，`GitWorkspace.MERGED != WorkTask.DONE`。  
因为工业 Agent 的状态真相不能被一句自然语言总结替代。

第三，`感知层` 在 AgentX 里已经被收口到 `ContextCompilationCenter`。  
当前代码真相是：

1. requirement / architect / coding / verify 都先走统一 context pack
2. 结构化事实优先进入 `FactBundle`
3. 非结构化补充当前仍以 lexical / symbol baseline 为主

而目标设计里，coding 阶段会进一步减少“平台预先喂代码片段”带来的幻觉，改成：

1. 平台继续给结构化事实
2. agent 自己用 Unix 类只读探索工具确认仓库真相
3. 读权限放宽，写权限继续严格受 `writeScopes` 约束

这背后的思路就是：  
代码真相优先靠真实探索确认，而不是靠平台提前筛一包 snippets 让模型误以为那就是真相全集。

第四，`决策层` 在 AgentX 里不是完全自由 agent，而是“固定 workflow + 局部 agent 决策”。  
LangGraph 只编排顶层固定节点；真正的任务派发、lease、heartbeat、recovery 继续放在 runtime。这样做的价值就是减少“自由推理把业务真相带偏”的风险。

第五，`动作层` 在 AgentX 里被严格限制。  
不是让模型自由拼任意 shell 或任意 HTTP，而是：

1. 工具白名单
2. command allowlist
3. `writeScopes`
4. blocker 升级链路
5. 未来接入外部资源时走 approval processing center，而不是让 worker 直接找人或直接扩权

这实际上是在动作层降低幻觉。  
因为很多工业事故不是“说错了一句话”，而是“错的判断被执行成了真实副作用”。

第六，`验证层` 在 AgentX 里也不是可选附属，而是主链闭环的一部分。  
当前口径已经明确是：

`merge-gate -> integration-test-gate -> verify`

也就是说，减少幻觉不只发生在“生成答案之前”，还发生在“交付结果之后”。  
如果测试证据、集成证据、verify 裁决不成立，系统就不应把结果当成 DONE。

所以如果让我把这题用一句话总结成 AgentX 版本，我会这样答：

“从学术上看，Agent 是目标、状态、感知、决策、动作和反馈的闭环；从工业上看，减少幻觉要靠结构化真相、受控工具、显式状态机、权限边界、验证闭环和人工升级；而在 AgentX 里，这些东西分别被固化成固定 workflow、ContextCompilationCenter、task/run/workspace 真相、tool contract、approval processing center 以及 merge/integration-test/verify 主链。” 

### [重要程度：高] 单 Agent 和多 Agent 应该怎么选？

这个问题非常高频。

最稳的原则不是“多 Agent 更高级”，而是：

如果一个 Agent 已经能在同一状态空间下稳定完成任务，就先不要拆。

单 Agent 更适合：

1. 目标单一
2. 状态集中
3. 工具不多
4. 评估口径统一
5. 不同环节之间耦合很强

它的优势是：

1. 系统简单
2. 状态管理容易
3. 调试成本低
4. 输出口径统一

多 Agent 更适合：

1. 职责明显不同
2. 上下文差异明显
3. 评估标准不同
4. 某些环节需要独立迭代

例如可以拆成：

1. 交互 Agent
2. 规划 Agent
3. 工具执行 Agent
4. 审核 / 安全 Agent

但多 Agent 的代价也很明显：

1. 状态同步更复杂
2. 通信成本更高
3. 调试和归因更难
4. 容易出现责任边界不清

所以面试里更稳的回答是：

“默认先单 Agent，把状态、工具和评估做扎实；只有当职责、上下文和评估口径明显不同的时候，再拆成多 Agent。”

### [重要程度：高] Agent 的记忆和状态应该怎么设计？

这个问题一定不要只答“短期记忆、长期记忆”。

更完整的说法是：Agent 需要的不只是 memory，而是显式状态设计。可以拆成四层：

1. 当前工作记忆
   - 当前窗口里正在处理的材料
2. 会话状态
   - 这一轮或这一段交互的进展
3. 任务状态
   - 任务到了哪个阶段、试过哪些动作、失败过什么
4. 长期记忆
   - 画像、偏好、历史经验、沉淀知识

这四层解决的问题不同：

`工作记忆`

1. 解决当前回答或当前动作
2. 生命周期很短

`会话状态`

1. 解决多轮连贯性
2. 生命周期是一次会话

`任务状态`

1. 解决流程推进和恢复
2. 生命周期通常跨多个回合

`长期记忆`

1. 解决个性化和长期复用
2. 生命周期较长

很多系统失败的原因是把这四类混在一起，全都塞进聊天历史。

更稳的系统设计是：

1. 把关键任务状态结构化存储
2. 把长期知识做检索或配置化
3. 把当前窗口只留给当下最相关的信息

一句话总结：

“Agent 最重要的不是记住更多，而是把不同类型的状态分层存好，用的时候再动态编译。”

### [重要程度：高] 多 Agent / 异步任务场景下，怎么防止上下文污染？如果结合 AgentX，你会怎么答？

这题很适合拿来区分“会聊 Agent”还是“做过真正可恢复的 Agent 系统”。

先定义什么叫上下文污染。  
在多 Agent、异步任务、并发执行场景里，上下文污染通常指这几类问题：

1. 一个 agent 看到了不属于自己职责范围的信息
2. 旧状态覆盖了新状态，或者新状态误读了旧状态
3. 别的任务的半成品、猜测结论、临时对话被当成正式真相
4. 人工回复、工具结果、检索片段、任务状态全部混进同一段 chat history
5. 同一个 run 重试时重复继承了错误的工作记忆

所以真正要防的，不是“上下文太长”，而是“真相边界失效”。

工业里更稳的做法一般有六条。

第一，按职责拆上下文，而不是多人共享一个大 memory。  
不同 agent 的目标、输入和决策权限不同，就不应该共享同一份完整上下文。例如规划 agent 关心的是 requirement、依赖和 blocker；coding agent 关心的是 task objective、write scope、工具和仓库证据；verify agent 关心的是 changed files、测试证据和交付裁决。这些内容混在一起，污染几乎是必然的。

第二，结构化真相和工作记忆分开。  
真正的任务状态、run 状态、workspace 状态要单独持久化，不能全靠聊天历史传递。聊天历史最多只承载某个 agent 当前这一轮需要处理的工作记忆，而不是系统总状态。

第三，交接靠显式 handoff，不靠隐式继承。  
多 agent 之间不要把整个对话原样转发，而应该交接结构化摘要，例如：

1. 当前任务目标
2. 当前状态
3. blocker
4. 已验证证据
5. 下一步允许动作

第四，快照要可追踪、可复用、可失效。  
上下文不是越新越好，而是要知道“这份包是基于什么真相编出来的”。如果底层事实变了，旧上下文应该过期，而不是继续参与后续决策。

第五，并发任务默认只共享摘要，不共享半成品原文。  
特别是在代码任务里，别的并发 task 尚未 merge 的中间代码不应该默认作为当前 agent 的输入，否则很容易把“别人的局部尝试”误当成“当前全局真相”。

第六，权限边界和上下文边界一起设计。  
如果一个 agent 根本没有权限操作某类资源，就不应该给它完整的动作语境，更不能让它通过读到别人的上下文，推导出自己可以越权行动。

落到 AgentX，这题可以直接讲成“结构化真相优先 + 节点化 pack + 并发摘要隔离”。

第一，AgentX 没有让所有 agent 共用一个 chat memory。  
当前 requirement、architect、coding、verify 都先走 `ContextCompilationCenter`，并且固定区分：

1. `workflow-scoped`
2. `task-scoped`

其中 requirement / architect 用 workflow-scoped；coding / verify 用 task-scoped。这个边界本身就是在防污染，因为架构阶段不会顺手吃进某个 task 的局部执行细节，coding 也不会把整条 workflow 的所有对话历史都背进来。

第二，AgentX 把系统真相放在结构化对象，而不是 memory。  
最关键的真相在：

1. `RequirementDoc`
2. `Ticket`
3. `WorkTask`
4. `TaskContextSnapshot`
5. `TaskRun`
6. `GitWorkspace`
7. `WorkflowRun / WorkflowNodeRun`

这也是为什么我们强调 `TaskRun.SUCCEEDED != WorkTask.DONE`。  
如果把这些真相塞回自然语言记忆里，异步恢复和多 agent handoff 很快就会失真。

第三，AgentX 用 context snapshot 防止“旧上下文继续污染新决策”。  
`CompiledContextPack` 自带：

1. `scope`
2. `sourceFingerprint`
3. `artifactRef`
4. `factBundle`
5. `retrievalBundle`

这意味着上下文不是匿名 prompt，而是可追踪快照。底层事实变化后，旧快照可以过期，不需要靠模型自己判断“我刚才看到的东西还算不算数”。

第四，AgentX 在并发代码任务里刻意不默认共享 sibling task 的原始半成品代码。  
当前和目标设计都强调：并发任务更适合共享结构化风险摘要，比如：

1. `taskId`
2. `status`
3. `writeScopes`
4. 最新 delivered / merged 状态
5. 潜在路径重叠风险

而不是直接把另一个 worktree 的未 merge 代码片段喂给当前 coding agent。  
这是一个很重要的抗污染设计，因为未 merge 半成品不是全局真相。

第五，AgentX 还用工作区和写域隔离降低上下文污染带来的副作用放大。  
每个 `TaskRun` 有独立 `GitWorkspace`，写入继续受 `writeScopes` 限制。即使上下文误判，错误也更容易被限制在当前任务边界内，不会立刻扩散成整仓副作用。

第六，人类介入也不走自由对话回填，而是统一走 `Ticket`。  
这相当于把“口头补充信息”转成了可追踪的结构化真相，避免某个 agent 看过一次人类回复，另一个 agent 却只继承到二手转述。

所以这题最稳的总结是：

“多 Agent / 异步任务要防的不是窗口长度，而是真相边界失效。最有效的办法是按职责拆上下文、把业务状态结构化、用快照和 fingerprint 管理上下文生命周期、并发任务只共享摘要不共享半成品，再用权限和工作区隔离兜底。AgentX 的做法就是 `ContextCompilationCenter + structured truth + task/workflow scoped pack + write scope / workspace isolation`。” 

### [重要程度：高] 如果要进行记忆压缩，通常有哪些方法？如果结合 AgentX，你会怎么做？

这题不要只答“做个 summary”，那样太浅。  
更稳的说法是：记忆压缩本质上是在做信息分层，把“当前必须保真的东西”和“以后按需再取的东西”拆开。

工业里常见的压缩方法通常有七类。

第一类，`滚动摘要`。  
把已经结束的一段对话或一组动作压成一段高密度摘要，只保留：

1. 已确认结论
2. 未解决问题
3. 下一步动作
4. 风险和约束

它适合压缩长对话，但不适合直接承担流程真相。

第二类，`结构化提取`。  
不是保留自由文本，而是把对话里的高价值信息抽成结构化字段，例如：

1. decision
2. blocker
3. task objective
4. acceptance criteria
5. owner
6. due / lease / status

这种方式比自然语言摘要更稳，因为后续可以精确查询和增量更新。

第三类，`快照 + 增量`。  
不要一直背全量历史，而是保留：

1. 最新状态快照
2. 自上次快照以来的 delta

这样比“每轮继承完整历史”更节省窗口，也更适合恢复和回放。

第四类，`分层摘要`。  
把原始信息分成：

1. turn summary
2. episode summary
3. task summary
4. project summary

这样越久远的信息越抽象，越临近当前动作的信息越细。

第五类，`证据外置 + 引用回链`。  
大体量日志、长文档、完整工具输出不需要每轮进窗口，可以外置成 artifact，只在记忆里保留：

1. artifact ref
2. 结论摘要
3. 关键证据点

需要时再按 ref 回查原文。

第六类，`去重与失效淘汰`。  
很多对话历史其实是重复确认、过期假设、旧路径探索结果。它们应该被：

1. 去重
2. 标记 superseded
3. 加 TTL
4. 超时淘汰

否则窗口会被低价值历史占满。

第七类，`检索化`。  
不是把所有历史硬塞进 prompt，而是把可复用历史沉淀成可检索资产，例如：

1. FAQ
2. 模块地图
3. 历史决策摘要
4. eval findings
5. verify 脚本索引

需要时按 query 再召回。

如果把这些方法压成一句话，就是：

“记忆压缩不是删信息，而是把原始历史转成摘要、结构化状态、快照增量和可检索 artifact。” 

如果结合 AgentX，我会强调三件事。

第一，AgentX 里最该被压缩的不是系统真相，而是自由对话和运行过程。  
像 `RequirementDoc`、`Ticket`、`WorkTask`、`TaskRun`、`GitWorkspace`、`WorkflowRun / WorkflowNodeRun` 这些对象，本来就应该直接进入结构化真相层，而不是先当 memory，再压缩。

第二，AgentX 里最适合做压缩的是“工作记忆层”。  
例如：

1. requirement 多轮补洞历史
2. architect 的讨论过程
3. coding 的多轮工具观察
4. verify 的阶段性分析

这些内容不适合全量长期挂在窗口里，更适合被压成：

1. 当前节点摘要
2. 最新 blocker
3. 最新证据 ref
4. 当前下一步建议

第三，AgentX 的长期复用层应该优先沉淀“决策和边界”，而不是沉淀整段原始对话。  
更值得长期保留的通常是：

1. 已确认 requirement 结论
2. 已批准 blocker 答案
3. 历史架构决策摘要
4. 公共模块地图
5. verify 失败模式与修复建议
6. eval artifact 与可复用经验

而不是一整段自由 chat transcript。

所以这题最稳的 AgentX 总结是：

“记忆压缩的核心不是把聊天记录缩短，而是把自由过程沉淀成结构化真相、节点摘要、artifact ref 和长期可检索知识。AgentX 里 requirement/ticket/task/run/workspace 不该被压成 memory；真正该压缩的是对话过程、工具过程和阶段性分析结果。” 

### [重要程度：高] 你怎么理解 Agent 的短期记忆和长期记忆？如果结合 AgentX，应该怎么落？

这题不要只答“短期记忆放窗口，长期记忆放向量库”，那样太粗。

更稳的说法是：短期记忆和长期记忆解决的是两类完全不同的问题。

先说短期记忆。  
短期记忆的核心作用不是“多记一点”，而是支撑当前局部决策。它通常包括：

1. 当前轮的任务目标
2. 最近几步工具观察
3. 当前还没解决的局部假设
4. 当前窗口里最相关的证据

它的特点是：

1. 生命周期短
2. 强依赖当前任务
3. 很容易过期
4. 适合跟着当前上下文包一起裁剪和重编译

再说长期记忆。  
长期记忆的核心作用不是支撑当前一步，而是提高未来复用率。它通常包括：

1. 用户偏好
2. 领域知识
3. 过往案例
4. 历史决策
5. 稳定规则和经验总结

它的特点是：

1. 生命周期长
2. 跨任务复用
3. 不一定每轮都该进窗口
4. 更适合做检索、索引或结构化沉淀

但工业里还有一个特别容易讲错的点：  
很多人把“长期记忆”说成系统里所有持久化数据，这其实不准确。

真正的业务真相，例如任务状态、工单状态、运行状态、权限状态，严格说不应该算 memory，而应该算 system of record。  
因为它们不是“记住了也行，忘了也行”的辅助信息，而是流程正确性的依据。

所以更完整的工程化拆法其实是三层：

1. 短期记忆
   - 当前决策窗口
2. 长期记忆
   - 可复用经验和稳定知识
3. 结构化真相
   - 流程状态、业务状态、权限状态、执行证据

如果结合 AgentX，这个边界要讲得更清楚。

第一，AgentX 当前真正落地得最重的不是 memory，而是结构化真相。  
像：

1. `RequirementDoc`
2. `Ticket`
3. `WorkTask`
4. `TaskRun`
5. `GitWorkspace`
6. `WorkflowRun / WorkflowNodeRun`

这些都不应该被叫成“长期记忆”，它们是固定主链的正式状态真相。

第二，AgentX 的短期记忆更接近当前节点的 `CompiledContextPack`。  
也就是 requirement / architect / coding / verify 每次真正拿到模型前的一包动态编译结果。  
这份包里会放当前最相关的：

1. facts
2. retrieval snippets
3. tool guardrails
4. write scopes
5. 最新 run / workspace 证据

它本质上就是节点级、任务级的工作记忆，而不是永久记忆。

第三，AgentX 的长期记忆目前更偏“可复用事实与经验资产”，而不是 memory-first 框架。  
当前已经比较接近长期复用层的内容包括：

1. repo 级索引
2. docs / schema / FAQ / interview bank 这类中间知识资产
3. 已落盘的 context artifact
4. workflow 运行证据和 eval artifact

未来如果继续往前走，`Historical Decision Reuse` 也会是长期记忆的一部分，但它依旧应该建立在结构化事实和可检索 artifact 之上，而不是把所有历史都塞回聊天窗口。

第四，AgentX 当前明确不走“统一 chat memory 解决所有问题”的路线。  
我们的核心是 `ContextCompilationCenter`，也就是：

1. 先取结构化真相
2. 再取当前真正需要的补充材料
3. 再按节点和 scope 编译成短期工作记忆

这比“挂一个长期 memory 容器让所有 agent 共用”更符合固定 workflow 平台的边界。

如果面试官继续追问“那到底什么信息放短期，什么信息放长期”，我会给一个判断标准，而不是背概念。

适合放短期记忆的，一般有四个特征：

1. 只服务当前节点或当前 run
2. 很容易过期
3. 需要高保真地支撑下一两步动作
4. 一旦任务切换，价值就会快速下降

典型包括：

1. 当前 task objective
2. 最近几步工具观察
3. 当前怀疑要改的文件和路径
4. 当前 blocker 和待确认问题
5. 最近一次 run / workspace 证据摘要

适合放长期记忆的，一般也有四个特征：

1. 会跨任务复用
2. 在未来仍然成立
3. 更像规则、经验、偏好或稳定知识
4. 不需要每轮都高保真原文进入窗口

典型包括：

1. 用户或团队偏好
2. 模块地图和公共组件认知
3. 历史架构决策摘要
4. 已验证的 verify 脚本和失败模式
5. 已批准的资源授权和 integration contract 摘要

但还有第三类必须单独说清楚：  
有些信息既不是短期记忆，也不是长期记忆，而是 `system of record`。例如：

1. `RequirementDoc`
2. `Ticket`
3. `WorkTask`
4. `TaskRun`
5. `GitWorkspace`
6. `WorkflowRun / WorkflowNodeRun`

这些内容如果被误放进“长期记忆”容器，面试里其实是减分项，因为它们不是“可记可不记”的知识，而是流程正确性的正式依据。

所以这题最稳的总结是：

“短期记忆服务当前决策，长期记忆服务未来复用，但两者都不能替代结构化真相。放到 AgentX 里，`CompiledContextPack` 更像短期工作记忆，docs/index/eval/historical artifacts 更像长期可复用知识，而 requirement / ticket / task / run / workspace 则是必须单独管理的系统真相。” 

### [重要程度：高] 你了解哪些 Agent 设计范式？如果结合 AgentX，应该怎么选？

这题不要背名词表，最好先讲“范式是为了解决什么约束”。

常见 Agent 设计范式大致可以分成六类。

第一类，`ReAct`。  
核心是 `reason -> act -> observe` 的局部循环。优点是灵活，适合开放探索；缺点是状态容易漂在过程里，副作用幂等和审计会比较难做。

第二类，`Plan-and-Execute`。  
先规划，再执行。优点是层次清楚，适合中长任务；缺点是计划和真实执行容易脱节，环境一变就要重规划。

第三类，`Manager-Worker`。  
一个上层 agent 负责拆解和分配，下层 agent 专注执行。优点是职责清晰，适合并发；缺点是 handoff、冲突治理和上下文边界设计不好时，很容易失真。

第四类，`Critic / Reflection`。  
先产出，再由审查者反思、批评或打回。优点是质量更稳；缺点是成本和延时更高，而且如果审查标准不结构化，也会变成两轮自由发挥。

第五类，`Workflow-driven Agent`。  
也就是流程节点固定，agent 只在局部节点里做决策。优点是稳定、可恢复、可治理；缺点是灵活性不如完全自由 agent。

第六类，`State-machine / Event-driven Agent`。  
把状态迁移、事件和恢复路径显式建模。优点是工程可控、适合异步和长任务；缺点是设计成本更高，对建模能力要求高。

面试里更高级一点的讲法，不是说哪种最好，而是说它们适合不同层级：

1. 局部问题求解
   - 更适合 `ReAct`
2. 中程任务拆解
   - 更适合 `Plan-and-Execute`
3. 多角色协作
   - 更适合 `Manager-Worker`
4. 高质量交付和安全兜底
   - 更适合 `Critic / Reflection`
5. 平台主链推进
   - 更适合 `Workflow-driven + State-machine-driven`

如果结合 AgentX，答案会更明确。

第一，AgentX 顶层不是纯 ReAct。  
我们没有让一个超级 agent 用自由 thought/action 回路推动全流程，而是冻结了固定主链：

`requirement -> architect -> ticket-gate -> task-graph -> worker-manager -> coding -> merge-gate -> integration-test-gate -> verify`

这说明 AgentX 的顶层更接近：

1. `Workflow-driven`
2. `State-machine-driven`

第二，AgentX 的任务执行层明显带有 `Manager-Worker` 范式。  
`架构代理` 负责拆 task 和重规划，`工作代理管理器` 负责选择 capability 和派发，worker 不自抢任务。这不是自由自治集群，而是中心派发制的 manager-worker 变体。

第三，AgentX 在 coding 局部回路里保留了受限的 `ReAct-like` 模式。  
coding agent 每轮仍然会根据观察决定：

1. `TOOL_CALL`
2. `ASK_BLOCKER`
3. `DELIVER`

但这不是开放式 ReAct，因为动作已经被 `ToolCall` 协议、`callId` 幂等、`writeScopes` 和 command allowlist 强约束了。

第四，AgentX 在交付闭环上带有 `Critic / Reflection` 思想，但没有把它做成自由辩论。  
也就是：

`merge-gate -> integration-test-gate -> verify`

verify agent 不是和 coding agent 自由辩论，而是读取结构化证据后做裁决。这样保留了“先做再审”的优点，又避免 critic 和 actor 无限对话。

第五，AgentX 的核心设计哲学其实是“顶层固定、局部自治、状态外置、证据优先”。  
这和很多 demo 型 Agent 系统很不一样。后者喜欢把所有能力堆成一个会思考的单体 agent；而 AgentX 更像一个把多种范式按层组合起来的平台。

所以这题最稳的总结是：

“Agent 范式没有银弹。ReAct 适合局部探索，Plan-and-Execute 适合中程拆解，Manager-Worker 适合多角色协作，Critic 适合质量把关；但像 AgentX 这种需要可恢复、可审计、可治理的平台，顶层更适合 `fixed workflow + state machine + central dispatch`，局部再嵌入受限的 ReAct-like 工具回路和 verify 裁决。” 

## 工具、RAG 与上下文

### [重要程度：高] Agent 为什么要有工具，而不是只靠模型本身？

因为很多真实任务不是语言问题，而是行动问题。

模型本身擅长：

1. 理解意图
2. 生成文本
3. 做有限推理

但它不天然擅长：

1. 查最新数据库
2. 调外部 API
3. 执行系统动作
4. 访问真实文件和日志
5. 进行高精度计算

工具的意义就是把“语言能力”接到“真实世界能力”上。

一个成熟的 Agent 系统里，工具通常承担三类任务：

1. 读
   - 查询资料、数据库、日志、文件
2. 算
   - 规则判断、数值计算、排序过滤
3. 做
   - 提交请求、创建记录、调用服务、执行动作

所以工具不是附属品，而是 Agent 从“会说”走向“会做”的关键桥梁。

### [重要程度：高] Agent 怎么做 RAG，才能尽量减少幻觉？

这题不要只说“接个向量库”。

更完整的答法是：

第一，先明确事实来源的优先级。  
不是所有来源都同权。应该先定义：

1. 哪些是可信真相
2. 哪些是辅助解释
3. 哪些只能做参考

第二，结构化事实优先。  
只要是核心业务结论、状态、规则、配置，不要完全依赖模糊召回，尽量走精确查询或结构化存储。

第三，RAG 只补模型当前不知道、但任务又需要的片段。  
不要把它做成全文灌输器。

如果是 coding agent，还要再补一句：  
代码任务不一定应该把 repo 代码主要通过 RAG 注入。很多时候更稳的做法是保留结构化事实层，再让 agent 用受控的 Unix 类工具自己确认目录结构、搜索范围和具体实现位置。

第四，要保留来源和证据。  
如果 Agent 输出关键结论，最好能说明：

1. 来源于哪个文档 / 记录
2. 哪些片段支撑了这个结论

第五，要有检索失败或证据不足时的降级策略。  
如果可信材料不够，宁可保守回答、追问澄清或升级人工，也不要强行编。

一句话总结：

“好的 RAG 不是让模型看更多，而是让模型优先基于更可信、与任务更相关的材料说话；而在代码任务里，很多代码真相最好通过工具探索来确认，而不是靠平台预先喂一包片段。”

### [重要程度：高] 当对话轮次过多、上下文窗口不足时，应该怎么处理？

正确优化方向不是继续堆文件，而是把上下文组织方式升级。

常见优化路径包括：

1. 用结构化状态替代一部分原文
2. 建立项目 / 业务中间层摘要
3. 对材料做分层索引
4. 按当前任务动态编译上下文
5. 优先保留事实，次优保留检索片段

更具体地说：

第一，不要每次都喂原始全文。  
很多高价值信息其实应该提前被整理成：

1. 模块地图
2. 调用关系摘要
3. FAQ
4. 关键规则说明
5. 文件职责索引

第二，把稳定知识和运行期增量分开。  
否则每轮都会重复读很多不变内容。

第三，做 query-aware context selection。  
根据当前任务的目标、路径、实体、约束去选材料，而不是靠人工拍脑袋塞一堆文件。

第四，把编译结果复用起来。  
如果输入没变，就不需要每次都重新编译整包上下文。

如果面试官继续追问“那长对话具体怎么处理”，我会再补一层实操策略，而不是只讲原则。

第一，`保留原文，但不保留在窗口里`。  
完整 transcript 可以外置保存，但进入当前窗口的只保留：

1. 当前节点摘要
2. 最新未决问题
3. 当前状态快照
4. 指向原始证据的 ref

第二，`只给未解决问题高保真原文`。  
已经解决的对话段落可以压成摘要；还在争议中的需求、刚失败的工具输出、当前待确认的 blocker，才值得保留更高保真的原文片段。

第三，`按节点重开上下文，而不是按会话无限续写`。  
很多系统一长就坏，是因为 requirement、architect、coding、verify 全部共用一条对话链。更稳的做法是按阶段切上下文，每次把上一阶段的结果 handoff 成摘要和结构化真相，而不是原样继承全部聊天记录。

第四，`用快照替代回放`。  
一旦任务状态已经被提炼成快照，就不需要每次再读完整历史推导“现在到哪一步了”。这会显著减少 token 浪费。

第五，`优先裁剪补充材料，不优先裁剪真相层`。  
窗口不够时，应该先裁：

1. 低相关检索片段
2. 重复解释
3. 旧搜索路径
4. 已过期假设

而不是先裁掉当前任务边界、权限边界和最新状态结论。

第六，`必要时做子线程 / 子任务 handoff`。  
如果一个问题已经长到不适合继续在原线程内推进，就应该生成一份 handoff summary，在新线程或新 agent 上继续做局部问题求解，而不是硬拖着整段历史一起跑。

如果结合 AgentX，我会直接把这个问题落到 `ContextCompilationCenter`。

第一，AgentX 不是把完整对话历史直接喂给模型。  
它的主路径是先拿结构化事实，再按 `workflow-scoped` 或 `task-scoped` 编译 `CompiledContextPack`。这本身就是在解决“轮次过多怎么办”。

第二，AgentX 会把高价值信息外置成可追踪 artifact。  
真正应该长期保留的是：

1. requirement truth
2. ticket / blocker 结论
3. task / run / workspace 状态
4. eval artifact
5. context artifact ref

而不是无限增长的 chat transcript。

第三，AgentX 的窗口裁剪原则应该是“先裁 retrieval，后裁事实”。  
这和项目里一直强调的 `structured-truth-first` 是一致的。窗口紧张时，宁可减少补充 snippets，也不要让模型丢掉 task 边界、write scope、当前 run 状态这些硬约束。

第四，AgentX 更适合“节点级重编译”，不适合“统一长记忆续写”。  
requirement / architect / coding / verify 每次都应该按当前节点重新取包。这样对话再长，也不会把整个系统拖进同一个无限膨胀的 memory 容器。

所以一句话可以概括为：

“窗口优化的本质不是压缩字符数，而是把原始信息转成可分层、可复用、可按任务装配的知识结构。放到 AgentX 里，就是结构化真相外置、节点级 pack 重编译、artifact 回链、窗口内只留当前最相关的一小层工作记忆。”

## 评估、治理与落地

### [重要程度：高] 如果让你做一个“提升大模型解题准确率”的探索型项目，你会怎么拆评测、归因和改进闭环？

这题不要一上来就答“加个工具”或者“改 prompt”。  
更稳的说法是：这种探索型项目的第一目标不是马上想方案，而是先把“问题长什么样”看清楚。

我会按四步拆。

第一步，`先建评测集和任务分层`。  
如果没有稳定评测，你根本不知道优化是不是成立。至少要把题目分层成：

1. 纯计算题
2. 应用题 / 文本题
3. 多步推理题
4. 需要外部知识或规则约束的题

同时保留：

1. 最终答案是否正确
2. 中间推理是否正确
3. 哪一步开始偏
4. 是否有格式、工具、停止条件问题

第二步，`系统化捞 badcase 并做错误归因`。  
不要只看总准确率，而要把 badcase 分成几类：

1. 读题理解错
2. 条件抽取错
3. 中间推理链断裂
4. 纯计算错
5. 工具该用没用
6. 工具用了但结果没接回主推理
7. 明明思路对，但变量绑定或状态跟踪错

只有把问题归因拆开，后面的优化才不会全靠猜。

第三步，`针对归因做有边界的改进`。  
可能的改进手段包括：

1. prompt / 输出结构约束
2. 先抽条件、再求解的工作流拆分
3. verifier / critic 二次校验
4. 外部计算工具或方程求解工具
5. few-shot exemplars
6. 题型分类后走不同 solve path

但每种改进都应该和某类错误直接对应，而不是“大家都试一遍”。

第四步，`回到评测闭环验证收益和副作用`。  
看提升不能只看单题案例，要看：

1. 总准确率是否提升
2. 哪类题提升
3. 哪类题被副作用拖差
4. 调用工具后的成本、时延和稳定性是否可接受

如果结合工程化，我会强调这其实就是一个“eval-driven optimization”问题，而不是单次拍脑袋试想法。

如果结合 AgentX，我会直接映射到我们项目已有的方法论。  
虽然 AgentX 当前主场景是软件交付，不是数学解题，但方法是共通的：

1. 先用 scenario pack 和评测集固定输入
2. 再用 badcase review 做人工归因
3. 把归因转成具体改进项
4. 最后回到统一评测中心看改动有没有真实收益

也就是说，AgentX 给这类探索项目最大的启发不是“某个 prompt 模板”，而是“先评测、再归因、再局部改动、最后用统一报告回归验证”这条闭环。

一句适合面试的总结是：

“提升大模型解题准确率，本质上不是先想技巧，而是先建评测和错误分类。先知道它错在哪，再决定该用 prompt、工作流、verifier 还是外部工具去改，最后回到统一评测闭环验证是否真的有效。” 

### [重要程度：高] 为什么大模型单独解一个算式可能是对的，但放进题目场景里就更容易错？如果要改进，你会怎么做？

这题很有代表性，因为它能区分“纯计算能力”和“题目求解能力”不是一回事。

更稳的理解是：模型在应用题里做的事情，不只是算，还包括：

1. 读题
2. 抽条件
3. 建变量
4. 选公式或策略
5. 维护中间状态
6. 再把计算结果接回语义结论

所以单独给它一个方程时，它只是在做“局部计算”；  
但把它放进题里，它就要同时处理“理解 + 表征 + 推理 + 计算 + 回写”这一整条链。  
链条一长，错误点自然更多。

我一般会把原因归成四类。

第一，`题意理解负担`。  
模型可能不是不会算，而是先把条件读错了。

第二，`变量绑定和状态跟踪负担`。  
应用题里常见的问题不是公式错，而是把“谁对应谁”“中间量代表什么”弄乱了。

第三，`推理链和计算链耦合过深`。  
如果模型一边想步骤、一边自己算，中间任何一步漂掉都会把后面带偏。

第四，`工具使用时机不清楚`。  
就算接了计算器，也还有一个关键问题：什么时候该用、什么时候不该用。如果这个触发条件模糊，系统一样不稳定。

所以改进也不应该只是“加一个计算器”。

更稳的做法通常是三层。

第一层，`把求解流程拆开`。  
先让模型：

1. 抽题目条件
2. 明确变量和目标
3. 确认解题路径
4. 最后才进入计算

第二层，`把精确计算外包给工具，但加清晰触发条件`。  
不是 1 + 1 也调工具，而是给系统一套更明确的使用边界，例如：

1. 多步数值运算
2. 方程求解
3. 分数 / 根式 / 概率类高出错计算
4. 模型过去 badcase 集中暴露的高风险计算类型

第三层，`用评测把触发阈值调出来`。  
这件事不能靠拍脑袋决定，而是应该基于 badcase 和题型分层去看：

1. 哪类题调用工具后收益明显
2. 哪类题调用工具只是增加时延
3. 哪类题主要问题根本不在计算，而在读题和抽条件

如果结合 AgentX 的思路，这和我们做 coding / verify 的方式其实很像：  
不是先假设某个工具万能，而是先看错误归因，再把工具接到真正需要它的环节上，并用评测去验证收益。

一句最稳的总结是：

“单独算式能做对，不代表题目求解也能做对，因为应用题多了读题、变量绑定、状态跟踪和结果回写这些额外负担。改进不只是接计算器，而是把求解流程拆开、给工具设置明确触发边界，再用评测验证它到底在哪些题型上真的有帮助。” 

### [重要程度：高] 你会怎么评估一个 Agent 系统做得好不好？

不要只答用户满意度，也不要只答模型准确率。

更完整的评估通常分三层：

第一层，系统层指标。  
例如：

1. 成功完成率
2. 任务时长
3. 工具调用失败率
4. 人工接管率

第二层，体验层指标。  
例如：

1. 满意度
2. 追问率
3. 中途放弃率
4. 错误抱怨率

第三层，业务层指标。  
例如：

1. 转化率
2. 留存
3. 成本节省
4. 人效提升
5. 核心业务目标提升

另外最好再做错误归因，把问题拆成：

1. 检索问题
2. 推理问题
3. 工具调用问题
4. 状态管理问题
5. 安全与策略问题

这样评估才可行动，而不是只停留在“大概好不好”。

### [重要程度：高] 如果 Agent 做错了，或者输出不稳定，应该怎么治理？

一个成熟的回答应该从“发现、隔离、回放、修复、预防”五步讲。

第一，发现问题。  
要能通过日志、反馈、监控、抽样评估及时发现错误。

第二，隔离问题。  
先判断问题属于哪一层：

1. 输入问题
2. 检索问题
3. 推理问题
4. 工具问题
5. 权限问题

第三，回放问题。  
把当时的输入、上下文、检索结果、工具调用、最终输出完整回放出来。

第四，修复问题。  
根据归因修：

1. prompt
2. 工具协议
3. 检索策略
4. 状态建模
5. 安全规则

第五，预防问题。  
把这类错误加入评测集、回归集或策略规则中，避免重复发生。

所以面试里可以总结成：

“Agent 治理的关键不是把错误藏起来，而是让错误可观测、可回放、可归因、可回归验证。”

### [重要程度：高] 你会怎么控制 Agent 的权限和安全边界？

Agent 一旦能调用工具和执行动作，就必须有显式权限边界。

常见做法包括：

1. 工具白名单
   - 只允许调用明确注册的工具
2. 参数约束
   - 限制输入格式、目标范围、资源范围
3. 动作分级
   - 读操作自动执行，写操作人工确认，高风险操作禁止
4. 审计日志
   - 保留调用链路和最终结果
5. 降级机制
   - 证据不足或不确定时，退回解释、追问或转人工
6. 资源审批中心
   - 对需要额外资源或外部系统访问的动作，不要每次同步打断，而是走规范化请求、异步审批和授权复用

安全边界设计的核心不是“尽量不让 Agent 做事”，而是：

“让它只能在明确授权、可监控、可回滚的范围里做事。”
